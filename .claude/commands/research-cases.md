**아래 [리서치 산출 설정]과 [상세검색 설정]을 먼저 확정해 `검색설정 블록`을 만든 뒤, 그 블록을 실행 프롬프트에 포함하여 `case-researcher` 서브에이전트를 실행하라.** (순서: 설정 확정 → 블록 생성 → 블록 실은 Task 호출)

- **[입력 접수 — 붙여넣기 표(기본)]** 실행 방식은 두 가지다. 어느 쪽이든 최종적으로 아래 필드를 모두 확정해 `검색설정 블록`을 만든다.
  - **(A) 인라인($ARGUMENTS)이 있으면** 그대로 파싱한다(빠른 경로). 구분자 `|` — 1번째=키워드, 2번째=사실관계, 3번째(선택)=검색설정 `키=값` 토큰. 부족한 상세검색 항목은 완전성 기본값으로 채운다. **단 키워드·사실관계·유용성방향성 중 빠진 게 있으면 (B) 표를 출력해 마저 받는다.**
  - **(B) 인라인이 비었으면(또는 필수 항목 누락) 아래 "설정 표"를 코드블록으로 그대로 출력하고 멈춰서 사용자의 붙여넣기를 기다린다.** `AskUserQuestion`을 쓰지 않는다. 안내 문구: "값만 채워 이 블록째 다시 붙여넣으세요(빈칸은 완전성 기본값)."

    ```
    # /research-cases 리서치 설정 — 값만 채워 이 블록째 붙여넣기
    # (# 뒤는 안내. 빈칸이면 완전성 기본값. 구분자는 = )
    검색키워드   =
    사실관계     =            # 1~3문장. 각 판례 유사성 판정 기준
    # ── 상세검색 ──
    모드         = 완전성      # 완전성 | 빠른탐색
    페이지상한   =            # 빠른탐색일 때만 (기본 10)
    사건유형     = 형사        # 형사 | 전체 | 민사 | (직접입력)
    정렬         = 관련도순    # 관련도순 | 최신순 | 인용순
    재판유형     = 전체        # 전체 | 판결 | 결정
    선고일       = 전체        # 전체 | 최근3년 | 최근10년 | YYYYMMDD-YYYYMMDD
    법원         = 전체        # 전체 | (법원명 직접입력)
    조건검색     =            # 예: 완전일치:고가, 제외:코인, 법조문:형법347
    주문유형     =            # 보통 비움
    # ── 산출물(docx) ──
    상위N        = 30         # 최종본에 담을 판례 수(나머지는 백업)
    유용성방향성 =            # "어떤 논증을 뒷받침하는 판례가 가장 유용한가"
    ```
  - **표 파싱 규칙**: 각 줄 `키 = 값`. `#` 뒤·빈 줄·`──` 구분선은 무시. **값이 비면 완전성 기본값** 적용(모드=완전성·사건유형=형사·정렬=관련도순·재판유형=전체·선고일=전체·법원=전체·주문유형 비움·조건검색 없음·페이지상한 없음(끝까지)·상위N=30). 표에 없는 축(탭=판례 등)도 기본값. 값이 애매하면(예: 알 수 없는 사건유형) **그 항목만** 사용자에게 되묻는다.
  - **여전히 비면 안 되는 것**: 붙여넣은 표에도 **키워드·사실관계·유용성방향성**이 비어 있으면 채워달라고 다시 요청한다(이 3개는 리서치·정렬의 축이라 필수).
- 사이트는 lbox. **도메인**(`https://lbox.kr`)을 쓴다. 시작 전 **로그인 상태를 먼저 확인**하고, 안 돼 있으면 멈추고 사용자에게 직접 로그인을 요청할 것(자격증명 입력·저장·추측 금지).

- **[산출 설정의 의미 — 정렬·bold 기준]**
  1. **최종본 상위 N**(표 `상위N`) — 최종 docx에 담을 판례 수(기본 **30**). 나머지 부합건은 백업으로만 기록.
  2. **유용성 방향성**(표 `유용성방향성`, 골드기준) — "어떤 논증을 뒷받침하는 판례가 가장 유용한가". 예: *"피고인이 불법촬영(카촬) 등 다른 불리한 정황이 있음에도, 피해자의 거부 발언·일부 응하는 정황 등을 근거로 준강간은 별도로 심신상실·항거불능을 인정하지 않아 무죄로 본 판례"*. 이 방향성이 Phase3 정밀 재평가·정렬(유용성순)과 bold 대상 선정의 기준이 된다.

- **[검색설정 블록 생성]** 확정한 값들을 `키=값` 정규형 `검색설정 블록`으로 정리해 서브에이전트 실행 프롬프트에 함께 전달한다. 선택지·값의 정본(SSOT)은 `case-researcher` 에이전트 정의의 **'상세검색 설정 카탈로그'** 절이다. 프리셋 참고 — **완전성**: `사건유형=형사 법원=전체 정렬=관련도순 재판유형=전체 선고일=전체 주문유형= 탭=판례 모드=완전성`(페이지상한 없음·끝까지). **빠른탐색**: `모드=빠른탐색 페이지상한=10 정렬=관련도순 사건유형=형사`(나머지 완전성과 동일 — "상위 N페이지만 → 누락 0 보장 불가"를 명시). 서브에이전트는 이 블록을 그대로 적용·기록하고 사용자에게 되묻지 않는다.
- **실행 전 권한 확인**: worklist는 `Write(data/**)`로 통째 재기록한다(Edit 미사용 — 다른 환경에서 권한 프롬프트로 흐름이 끊기지 않게).
- **무중단 연속 실행(기본)**: 배치 사이·Phase1↔Phase2↔Phase3 사이에 **멈추지 않고 사용자 지정 상한까지 한 번에** 진행한다(승인 대기 게이트 없음). 컨텍스트 관리를 위한 자체 배치 분할·fresh 재개는 허용(그건 멈춤이 아님). 멈추는 건 ⓐ상한 도달 ⓑ차단 징후 ⓒ로그인 필요, 또는 사용자가 "테스트로 Phase1까지만" 등 **명시적으로 중간 정지를 지시**한 경우뿐.
- **속도(배치1 지연 교정)**: 필터·정렬·페이지 이동 등 반복·재렌더 요소는 **`role=`/텍스트 셀렉터로 바로 클릭**(고정 ref 재사용·ref 찾기용 재스냅샷·클릭 재시도 루프 금지). 페이지당 스냅샷 1회. 필터 적용 확인은 총건수 텍스트만으로.
- **★페이지네이션(치명버그 교정, 필수)**: lbox 번호형은 『숫자 5개 창+화살표』이고 **`다음으로 이동`은 1페이지가 아니라 +5(그룹) 점프**다(실측). **단일 페이지는 숫자 버튼(`role=button[name="57"]`)으로만** 이동하고, '다음으로 이동'은 다음 5개 창을 드러낼 때만 사용한다. 창 전환마다 보이는 숫자 창 첫 값이 기대값과 같은지 검증(어긋나면 중단·보고). 상세는 case-researcher 정의 §1-2 [A]. 페이지를 건너뛰면 인벤토리가 표본이 되어 완전성이 깨진다.
- **Phase1(목록 트리아지)**: 전달한 `검색설정 블록`을 적용(완전성 기본=법원 전체·모든 심급, 주문유형 비움, 정렬 관련도순)해 (상한까지) 캡처하고 `data/cases/{슬러그}/_worklist.md` 인벤토리를 완성. 죄목태그 1차 필터는 false-positive 우호(애매·태그무는 통과·보류; **단 사용자 특수필터가 있으면 그 지시 우선**). **정렬=관련도순은 비결정적이므로 완전성 검증(행수=총건수, 모든 심급)을 엄격 적용.** 총건수가 화면에 없거나 빠른탐색(상위 N)이면 "누락 0 보장 불가"를 worklist·최종보고에 명시. 검증 후 **곧바로 Phase2로 이어감**.
- **Phase2(전문 수집·분석·저장)**: 각 통과·보류 판례를 **`/case/{법원}/{사건번호}` 직접링크로 열어** `browser_evaluate(()=>main.innerText, filename="data/cases/{슬러그}/raw/{사건번호}.md")`로 **원문을 파일에 덤프**(모델 토큰 0 — 전문을 컨텍스트로 읽지 말 것). 대량(수백 건)이면 **수집(직렬, 단일 브라우저)과 오프라인 분석(병렬 워크플로)을 겹쳐** 돌린다(case-researcher §Phase3-0 파이프라인; 메인 세션이 수집 에이전트 ∥ 분석 워크플로를 동시에 오케스트레이션. **서브에이전트는 다른 에이전트를 스폰하지 않음**). 분석은 각 원문을 정독해 사실관계 유사성·부합 발췌·확정여부·심급을 판정·파일 저장. 확정여부·동일사건 심급은 `/case/` 페이지의 **'상•하위 판결 N [확정라벨]'** 영역에서 확인. (소수면 결과목록 카드클릭→인페이지 패널 방식도 가능.) 끝나면 Phase3로 이어감.
- **Phase3(최종 산출물 docx)**: 부합 판례 중 **유용성 상위 N건**만(§리서치 산출 설정) `.claude/style_case_research.docx` 템플릿 표로 만든다. **유용성 정렬 = ①유용성 방향성 부합도(최우선·최종 기준) → ②확정 여부 → ③심급(지방<고등<대법)**. **★선별된 상위 N건은 §3-2b '상급심 확정내용 검증'을 거친다** — "사건 확정"과 "내가 인용할 판시의 확정"은 다르므로(하급심 판시가 상급심서 파기돼도 사건은 '확정' 표시 가능), 인용할 판시가 상급심서 파기·변경 없이 그대로 확정됐는지 **상급심 원문(`/case/`)으로 확인**해 확정여부 칸·정렬·행 구성에 반영(파기·변경 시 강등·교체·제외, 불명확 시 `〔인용부 확인필요〕`). 전건 아님·상위 N건+버퍼 한정. **판례번호 셀 = 순수 형식 + lbox `/case/` 하이퍼링크**(클릭 열람), **내용 칸 = 【사실관계】+【법원의 특징적 설시】에 핵심만 bold**(전형적 '증명부족/무죄' 문구 제외). 산출: `output/{슬러그}_판례리서치_최종.docx`(상위 N) + 전체 부합 백업 docx + `_final_ranking.json`(전체 랭킹). 상세 규약·스크립트는 case-researcher §Phase3 및 `scripts/` 참조.
- 페이싱·차단회피·로그인·실검증의 정본은 `docs/lbox-pacing-and-verification.md` 준수: 클릭 위주, 직접 fetch 금지, 목록 복귀는 **상세 패널 닫기(닫기/Escape) 우선**(전문이 인페이지 패널이라 페이지 이동 없이 복귀), 동작 사이 가변 텀, 차단 징후 시 worklist Write 후 중단·보고. **저장/worklist 포맷의 정본은 case-researcher 에이전트 정의**(충돌 시 에이전트 본문 우선).
- 끝에 총건수(또는 관찰 행수)·심급 분포·통과/보류/제외 분포·저장 사건번호 목록·완전성 보장 여부·미해결 〔확인 필요〕·재개 지점을 보고할 것.
