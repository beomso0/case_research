`case-researcher` 서브에이전트를 실행하라.

- 검색 키워드 / 사실관계: $ARGUMENTS  (구분자 `|` 앞=키워드, 뒤=사실관계. 사실관계 없으면 사용자에게 1~3문장 요청)
- 사이트는 lbox. **도메인은 실검증에서 확정된 값**을 쓴다(미확정이면 0단계에서 라이브 관찰). 시작 전 **로그인 상태를 먼저 확인**하고, 안 돼 있으면 멈추고 사용자에게 직접 로그인을 요청할 것(자격증명 입력·저장·추측 금지).
- **실행 전 권한 확인**: worklist는 `Write(data/**)`로 통째 재기록한다(Edit 미사용 — 다른 환경에서 권한 프롬프트로 흐름이 끊기지 않게).
- **Phase1(목록 트리아지)**: 법원=전체(모든 심급)·기간=전체·주문유형 비움·정렬=결정적(안정)정렬 고정으로 검색결과를 끝까지 캡처해 `data/cases/_worklist-{슬러그}.md` 인벤토리를 완성. 죄목태그 1차 필터는 false-positive 우호(애매·태그무는 통과·보류). 완전성 검증(행수=총건수, 모든 심급) 후 보고하고 Phase2 승인 받을 것. 총건수가 화면에 없으면 "누락 0 보장 불가"를 명시 보고.
- **Phase2(전문 정독·저장)**: 통과·보류 행의 전문을 정독해 사실관계 유사성을 판단하고, 관련·보류는 **원문 전체 무손실**로 `data/cases/{사건번호}.md` 에 저장(frontmatter+§0 슬라이스 포인터+§1 전문). (a)~(d) 발췌·볼드는 하지 않고 §0이 위치만 지목. 매 건 worklist 즉시 Write.
- **Phase3(원문 발췌·볼드)·Phase4(docx 표)는 이 커맨드 범위 밖**(다음 단계). 심급관계·결론변경 row 분리도 Phase4에서 확정.
- 페이싱·차단회피·로그인·실검증의 정본은 `docs/lbox-pacing-and-verification.md` 준수: 클릭 위주, 직접 fetch 금지, 목록 복귀는 뒤로가기 우선, 동작 사이 가변 텀, 차단 징후 시 worklist Write 후 중단·보고. **저장/worklist 포맷의 정본은 case-researcher 에이전트 정의**(충돌 시 에이전트 본문 우선).
- 끝에 총건수(또는 관찰 행수)·심급 분포·통과/보류/제외 분포·저장 사건번호 목록·완전성 보장 여부·미해결 〔확인 필요〕·재개 지점을 보고할 것.
