# lbox 차단 회피 플레이북 + 실검증 프로토콜

> 대상: `case-researcher` 에이전트(신설 예정)와 이 에이전트를 오케스트레이션하는 메인 세션, 그리고 **사용자(변호사)**.
> 목적: lbox.kr(로그인 필요·유료·안티봇 작동)에서 **차단 없이, 사람처럼, 끝까지 누락 없이** 판례를 수집하기 위한 (a) 페이싱 규약과 (b) 키워드 1개로 돌리는 소규모 실검증 절차.
> 전제: 수집기(브라우저)와 정리/문서화(오프라인 docx)는 분리한다. 이 문서는 **수집기 절반(Phase1 목록 트리아지 + Phase2 전문 정독·저장)**만 다룬다. Phase3(원문 발췌+볼드)·Phase4(docx 표)는 저장 포맷이 지원하되 이번 범위 밖.
> 안티봇 한계: lbox.kr / support.lbox.kr 전 페이지가 비브라우저 접근에 HTTP 403. 아래 화면구조 관련 값 중 다수는 검색 스니펫 기반 간접 확인이므로, **정확한 셀렉터·페이지네이션·라벨·심급링크는 실검증(§3)에서 라이브 `browser_snapshot`으로 1차 확정한 뒤 에이전트에 하드코딩**한다.

---

## 실검증 확정값 (2026-06-29)
- 도메인 = `https://lbox.kr` (lbox.co.kr 연결거부).
- 검색 = 홈 입력창에 `"키워드"`(따옴표=필수 포함, `-`=제외) + Enter → `/task/{id}` 판례 결과.
- 결과 = 판례 탭 / 총건수 표시("검색결과 N개") / 정렬 기본 관련도순 / 번호형 페이지네이션(~10건/페이지) / 카드에 죄목 `[...]`·주문·인용수(피인용)·조회수 / 상세 `/precedent/{법원}/{사건번호}`.
- 잔여 미확인(전문 정독 배치에서 확정): 확정여부 라벨 위치·문자열, 전문 snapshot 추출/가상스크롤 누락, 동일사건 심급 링크.

## A. 차단 회피·페이싱 플레이북

lbox의 최우선 제약은 "비정상 접근(크롤링 오인) 차단"이다. 탐지는 (1) 요청이 너무 빠름·등간격, (2) 중간 단계 건너뜀(직접 URL 점프), (3) 즉시·기계적 클릭, (4) 같은 계정 동시 다중접속을 본다. 아래 6규약으로 이를 회피한다.

### A-1. 클릭 위주 네비게이션 (직접 URL fetch 최소화)

- **모든 이동은 화면 클릭으로 한다.** 검색 실행, 페이지 넘김, 판결 열기는 전부 `browser_snapshot` → 해당 요소의 `[ref=eNN]` 확인 → 그 ref로 `mcp__playwright__browser_click`.
- **`browser_navigate`(직접 URL 입력)는 다음 1회만 허용**: 작업 시작 시 lbox 홈(`https://lbox.kr/`) 진입. 그 외에는 검색결과·판결상세 URL을 추측·조합해 점프하지 않는다. (중간 단계 스킵은 그 자체로 탐지 신호이고, §0의 403이 보여주듯 lbox는 사람 브라우저 외 접근을 막는다.)
- 검색은 홈/검색창에 키워드를 `browser_type`으로 입력 후 Enter(`browser_press_key`) 또는 검색 버튼 클릭. URL `?query=` 조합으로 진입하지 않는다.
- ref는 **그 스냅샷 1장 안에서만 유효**하다. 페이지가 바뀌면 새 스냅샷을 떠 새 ref로 클릭한다. "기억해 둔 ref" 재사용 금지.
- **`browser_run_code_unsafe`·임의 JS 실행·`browser_network_request`로 직접 API 호출 금지.** 우회·핑거프린트 변조·stealth·프록시도 쓰지 않는다(약관·계정정지 리스크). 정답은 "사람처럼 느리게, 클릭으로, 양을 나눠서".

### A-2. 요청 간 간격 (가변 텀)

- 각 클릭(페이지 이동·판결 열기) 직후 `mcp__playwright__browser_wait_for`로 **실제 로드 완료를 기다린다**(특정 텍스트 출현 또는 목록/본문 요소 등장 대기). 로드 전 다음 동작 금지.
- 동작 사이 텀은 `browser_wait_for`로 실제 로드 완료를 기다리는 것으로 충분하다(Playwright 자체 대기). 추가 인위적 대기는 두지 않거나 **~0.5초**로 최소화한다(차단·로딩 오류 시에만 점진 증가). 고정 등간격 연타만 피한다(약간의 무작위성 유지). (Playwright MCP에 sleep 도구가 없으면 `browser_wait_for`의 `time` 인자 또는 "다음 콘텐츠 출현 대기"로 자연 텀을 만든다. 고정 1초 간격 연속 클릭 금지.)
- 페이지네이션·연속 열람이 길어질수록 텀을 약간 더 길게 가져간다.

### A-3. 세션당 처리량 상한

- **한 번의 에이전트 호출에서 여는 판결 전문 수에 상한**을 둔다. 기본값: 실검증 단계 **3~5건**, 정상 운용 **15~25건**. 도달하면 worklist를 저장하고 **정상 종료** → 다음 호출에서 재개(배치 운용).
- Phase1 목록 트리아지(카드만 읽고 기록)는 전문 열람이 아니므로 위 상한과 별개지만, 페이지 넘김도 §A-2의 텀을 지킨다.
- 시간당 전문 열람은 보수적으로 **수십 건 이하**로 시작. 정확한 임계는 미확인이므로 "이상 징후가 보이면 즉시 중단"을 절대 규칙으로 둔다(§A-4).
- 브라우저는 **단일 로그인 세션을 가진 공유자원**이다. 수집 서브에이전트는 **동시에 1개만**(병렬 금지). 영속 프로필을 두 인스턴스가 동시에 잡으면 프로필 락 충돌·세션 간섭이 나고, 같은 계정 동시접속은 lbox 쪽 차단 신호다. 오프라인 정리(Phase3·4)만 병렬 허용.

### A-4. 차단 징후 감지 → 즉시 멈추고 보고 (절대 규칙)

다음 중 **하나라도** 나타나면 더 진행하지 말고, ① worklist를 즉시 저장하고 ② 사용자에게 보고하고 ③ 작업을 멈춘다. 추측으로 우회하거나 재시도를 반복하지 않는다.

- 캡차/로봇확인 화면, "자동입력 방지" 류 챌린지
- "비정상 접근 / 이용 제한 / 일시 정지" 안내 문구
- 갑작스런 로그아웃, 로그인 페이지로의 강제 리다이렉트
- HTTP 403/429, 서버 오류 페이지
- **휴대폰 본인인증(카카오톡 인증번호) 요구 팝업** — 신규 기기로 오인된 신호. 절대 인증 시도하지 말고 보고.
- 빈 결과의 연속, 로딩 무한, 평소보다 현저히 느린 응답

보고 양식(사용자에게 전달):
```
[중단 보고] 차단 의심 징후 감지
- 징후: <캡차 / 403 / 강제 로그아웃 / 본인인증 팝업 / ...>
- 발생 시점: worklist에서 마지막 정상 처리 항목 = <사건번호 / 페이지 N>
- 현재 화면: <스냅샷 요지 1~2줄>
- 저장 상태: worklist 저장 완료(경로). 재개 가능.
- 권고: 일정 시간(예: 수십 분~수시간) 대기 후 재시도 / 사용자 직접 로그인 점검
```

### A-5. 과도한 비효율 발생 시 별도 보고 (사용자 요구)

차단이 아니더라도 **효율이 비정상적으로 나쁠 때**도 멈추고 보고한다(사용자의 명시 요구). 트리거 예:

- 한 판결 전문을 한 스냅샷으로 못 잡아 스크롤·재시도를 여러 번 반복(예: 한 건에 5회 이상 스냅샷)
- 페이지네이션이 끝없이 길거나(예상 대비 총 건수가 비현실적으로 큼) 무한스크롤이 끝을 안 보임
- 같은 항목이 반복 등장해 진척이 정체

보고에 ① 무엇이 비효율인지 ② 지금까지 처리량/잔여량 ③ 대안(검색어 정밀화 `@필드` 문법, 필터 좁히기, 배치 분할, Phase3로의 전환 등)을 함께 제시하고 사용자 판단을 받는다.

### A-6. 인지·토큰 규율 (차단 회피와 직결)

차단 회피의 핵심은 "느리게 + 적게 요청"이고, 그 전제는 **컨텍스트에 전문을 쌓아두지 않는 것**이다.

- **인지는 `browser_snapshot`(접근성 트리) 우선**, `browser_take_screenshot`은 예외(트리에 텍스트로 안 잡히는 배지/아이콘 확인 1장 한정). 확인하면 그 값을 즉시 worklist/파일에 텍스트로 옮겨 적고 스크린샷은 더 참조하지 않는다.
- **"스냅샷은 일회용 인지, 영속 기록은 파일"** — 목록 한 페이지를 스냅샷하면 필요한 필드(사건번호/제목/죄목태그/링크)만 즉시 worklist에 적고 다음 페이지로. 전문 1건을 스냅샷하면 **전문 전체를 §1에 원문 그대로 Write**하고(잘라내지 않음) §0 포인터·frontmatter를 채운 뒤 다음 건으로. 컨텍스트에 전문이 2건 이상 동시에 떠 있지 않게 한다(스냅샷은 누적되어 매 턴 다시 읽혀 토큰이 폭발한다).

## B. 로그인 절차

### B-1. 시작 시 로그인 상태부터 확인 (최우선)

1. `browser_navigate`로 lbox 홈(도메인 `〔실검증서 확정〕` — 실검증 0단계에서 라이브 관찰, 확정 후 하드코딩) 진입(시작 1회 허용된 직접 이동).
2. `browser_snapshot`으로 로그인 상태 판정:
   - 로그인 버튼/로그인 페이지/"로그인"·"회원가입" 링크가 보이거나, 마이페이지·로그아웃·계정명 등 **로그인된 표시가 없으면 = 미로그인**으로 간주.
3. **미로그인이면 즉시 멈추고** 사용자에게 다음과 같이 요청한다(작업 진행 금지):
   ```
   [로그인 필요] lbox에 로그인되어 있지 않습니다.
   열려 있는 브라우저 창에서 직접 로그인해 주세요. 완료되면 "로그인했다"고 알려주시면 이어서 진행합니다.
   (자격증명은 제가 입력·저장하지 않습니다. 로그인은 항상 사용자가 직접 합니다.)
   ```
4. **자격증명(아이디/비밀번호)을 입력·저장·추측하지 않는다.** `browser_type`으로 로그인 폼을 채우지 않는다. 이는 CLAUDE.md 강제 규칙.
5. 작업 도중 만료가 감지되면(§A-4의 강제 로그아웃/리다이렉트) worklist 저장 후 동일하게 멈춤·보고하고 사용자에게 재로그인을 요청한다.

### B-2. 본인인증 팝업 처리

- 5기기(PC 3 + 모바일 2, 브라우저 쿠키 기준) 초과 시 **휴대폰 본인인증**을 요구할 수 있다. 이 팝업이 뜨면 **인증 시도 금지**, 즉시 멈추고 보고(§A-4). 원인은 대개 "새 브라우저 = 신규 기기 오인"이며, 해결책은 §B-3의 영속 프로필이다.

### B-3. `.browser-profile` 영속화 (이번엔 사용자 동의 후 적용)

매 실행마다 새 브라우저 컨텍스트를 띄우면 lbox가 **신규 기기**로 계산 → 5기기 소진 + 본인인증 트리거 → 차단·중단 위험. 영속 프로필을 쓰면 **항상 같은 쿠키 = 같은 기기**로 인식되고 재로그인도 불필요해진다.

- **적용 방법**(`.mcp.json`의 playwright args에 추가):
  ```jsonc
  {
    "mcpServers": {
      "playwright": {
        "command": "npx",
        "args": ["-y", "@playwright/mcp@latest", "--user-data-dir", "./.browser-profile"]
      }
    }
  }
  ```
- **주의**: 단일 프로필 = **단일 순차 운용** 전제(§A-3). 두 인스턴스 동시 사용 불가(프로필 락).
- `.browser-profile/`은 로그인 세션이 담기므로 **절대 커밋 금지**(이미 gitignore 대상이어야 함).
- **이번에는 사용자 동의를 받은 뒤에만** `.mcp.json`을 수정한다. 동의 전에는 현재 설정 그대로 실검증을 돌리되, 본인인증 팝업이 뜨면 §B-2대로 멈춘다. 실검증 시작 안내에 "영속 프로필을 적용할지" 물어 동의를 받는 것을 권장.

## C. 실검증 프로토콜 (키워드 1개 소규모 시연)

목적: 본격 수집 전에 **작게(소수 건만)** 돌려, ① 차단 없이 돌아가는지 ② 화면구조 미확인 항목(7개)이 실제로 어떤지 ③ 저장 포맷이 Phase3/4를 받쳐주는지를 확인한다. **이번 프로젝트의 시작점이 바로 이 검증이다.**

### C-1. 사용자가 준비/입력할 것

사용자는 다음 3가지를 제공하고, 브라우저에 **직접 로그인**해 둔다.

1. **검색 키워드 1개** (예: `사기` 또는 더 구체적으로 `보이스피싱 사기` 등). 너무 광범위하면 결과가 폭주하니 검증용으로는 적당히 좁은 1개 권장.
2. **간단한 [사실관계] 1~3문장** — 트리아지·정독에서 "유사성" 판단 기준이 된다. 예: "피고인이 타인을 기망해 금원을 편취. 공범 없이 단독, 피해액 약 ○○원, 초범."
3. **죄목 키워드(목록 트리아지용)** — 카드의 [죄목] 태그에서 이 키워드가 없으면 1차 제외 기준. 예: `사기`, `사기방조`. (사실관계 키워드와 같을 수도, 다를 수도.)
4. **lbox 로그인 완료** + (선택) 영속 프로필 적용 동의 여부.

> 입력 양식 예(메인 세션/슬래시 커맨드에 전달):
> `키워드="보이스피싱 사기" / 사실관계="단독범, 기망 편취, 초범" / 죄목키워드=[사기,사기방조] / 검증건수=3`

### C-2. 에이전트의 시연 절차 (소규모)

**0단계 — 게이트.** §B-1 로그인 확인. 미로그인이면 멈추고 요청. (영속 프로필 미적용 시 본인인증 팝업 경계.)

**1단계 — 검색 + 완전성 필터 세팅.**
- 키워드를 검색창에 `browser_type` → 검색 실행(클릭/Enter). 직접 URL 금지.
- 필터를 **완전성 지향**으로 설정: 법원=전체(모든 심급), 기간=전체, 정렬=고정(기본값 그대로 두되 무엇인지 기록). 형사면 주문유형 필터는 검증 단계에선 **걸지 말고** 전체를 본다(누락 방지).
- `browser_snapshot`으로 **총 검색건수·페이지네이션 방식**을 확인해 worklist 헤더에 기록.

**2단계 — Phase1 목록 트리아지(소규모).**
- 검증에서는 **첫 페이지(또는 첫 1~2페이지)만** 트리아지한다(전수 아님 — §D 안전장치).
- 각 카드의 사건번호/제목/죄목태그/심급/lbox링크를 worklist 인벤토리 표에 기록.
- P1판정: 죄목태그에 죄목키워드 없으면 `제외(태그무)`. **태그 자체가 비어있으면 `보류(태그무)`로 두고 Phase2 대상에 포함**(false negative 방지).
- 페이지네이션 방식(번호/더보기/무한스크롤)을 **실제로 1회 동작시켜 판별**하고 worklist에 기록(완료 판정 로직의 핵심 변수).

**3단계 — Phase2 전문 정독·저장(검증건수만큼만).**
- 인벤토리에서 `P1=통과 또는 보류`인 행 중 **상위 N건(=검증건수, 기본 3건)**만 카드 **클릭**으로 상세 진입(직접 URL 금지).
- 각 상세에서 `browser_snapshot`으로 인지하되, **저장은 전문 전체를 §1에 원문 그대로 보존**한다. (a)당사자/(b)범죄사실/(c)법리/(d)적용판단은 **이번에 발췌·오려내지 않고** §0 슬라이스 포인터로 §1 안의 위치만 지목한다(발췌·볼드는 Phase3, 이번 범위 밖).
  - **확정여부 라벨**(열람페이지 좌측 상단) — 라벨 **문자열 원문 그대로** 기록. 위치·표현 추측 금지, 트리에 안 잡히면 스크린샷 1장으로 확인 후 텍스트로 옮김.
  - **lbox_url**: 카드/상세에 **실제 존재하는 href만** 기록. 없으면 `〔확인 필요〕`(URL 추측·조합 금지).
  - 유무죄, 심급, 선고일도 화면 문자열만.
- 동일 사건 1·2·3심은 **심급그룹키(SG-NNN)**로 묶음. Phase2 단건 정독 시점엔 상세의 관련사건 링크·주문 문구로 **잠정** 표기(상소심존재/파기 등). **결론변경 여부의 최종 판정과 docx row 분리는 Phase4(또는 별도 묶음 단계)에서 같은 심급그룹키 파일들을 모아 비교해 확정**한다.
- 각 건을 `data/cases/{사건번호}.md`에 §저장 포맷(SSOT=에이전트 본문 §2-3)으로 Write. P2판정·저장파일 경로를 worklist에 갱신.
- 동작 사이 §A-2 텀 준수. N건 도달 시 worklist 저장 후 정상 종료.

> **포맷 SSOT 주의**: 저장 파일·worklist의 **컬럼/구조 정본은 `case-researcher` 에이전트 정의(§2-3 저장 포맷·§worklist 양식)**다. 이 문서(docs)는 **페이싱·로그인·차단회피·실검증 절차의 정본**이고, 아래 두 부록은 에이전트 본문과 동기화된 요약본일 뿐이다. 충돌 시 에이전트 본문을 따른다.

**4단계 — 미확인 7개 항목 확정 보고.** 아래 체크리스트(§C-4)를 실제 관찰값으로 채워 사용자에게 보고.

### C-3. 성공 판정 기준

다음을 **모두** 충족하면 검증 성공:

1. **목록 캡처 완전성**: 트리아지한 페이지의 카드가 빠짐없이 worklist 인벤토리에 기록됨(페이지 내 항목 수 = 표 행 수). 페이지네이션 방식이 판별됨.
2. **worklist 생성**: `data/cases/_worklist-{키워드}.md`가 헤더(총건수/필터/페이지수) + 인벤토리 표로 정상 생성됨.
3. **전문 정상 저장**: 검증건수(예 3건)가 `data/cases/{사건번호}.md`로 저장되고, **§1 전문 무손실 + §0 슬라이스 포인터** + 확정여부 라벨 + lbox_url이 채워짐(발췌·볼드는 Phase3, 이번 미수행). 스냅샷에 **전문 텍스트가 실제로 담겼는지** 확인됨(캔버스/이미지 렌더링이 아님).
4. **차단 無**: 캡차·403·강제 로그아웃·본인인증 팝업·연속 빈결과 없이 완주.
5. **포맷 충분성**: 저장된 .md가 §1 전문 무손실 + §0 슬라이스 포인터 구조이고, Phase3(§1 원문에서 §0이 지목한 구간을 인용블록으로 오려 볼드만 추가) / Phase4(frontmatter의 사건번호+lbox_url→하이퍼링크, 유무죄, 확정여부, 심급그룹키 + 관련부분 컬럼=§0 포인터+§1 (b)(d))에 **추가 가공 없이 얹을 수 있는** 구조임.

하나라도 미달이면: 무엇이/왜 실패했는지 + 관찰값 + 권고를 보고하고, 에이전트 정의/필터/포맷을 보정한 뒤 재검증.

### C-4. 실검증에서 반드시 확정할 미확인 7개 (체크리스트)

라이브 관찰값으로 채워 보고. 이후 에이전트에 하드코딩.

- [ ] ⓪ **도메인**: 실제 접속·리다이렉트되는 정확한 홈 도메인(예 lbox.kr / lbox.co.kr). 확정 후 전 파일 통일.
- [ ] ① **페이지네이션 방식**: 번호 클릭 / "더보기" 버튼 / 무한스크롤 — 무엇이며 "끝 도달"을 어떻게 판정하나? (뷰포트=`browser_resize` 고정값도 함께 정함)
- [ ] ② **정렬 옵션·기본값·안정성**: 옵션 목록과 기본 정렬. **결정적 정렬(선고일/사건번호 등)이 있는지**, 그리고 **페이지를 넘겨도 카드 순서가 안정적인지**(정확도순은 동점 스코어로 순서가 흔들려 경계 누락 위험 → 차선). 완전성용 고정값 확정.
- [ ] ③ **확정여부 라벨**: 위치(좌측 상단 추정)·정확한 문자열(예 "확정"/"미확정"/"상고심 계속"). 텍스트인지 이미지인지.
- [ ] ④ **동일사건 심급 링크**: 상세에 "관련사건/상·하급심" 영역이 있나? 없으면 사건번호 패턴으로 수동 연결.
- [ ] ⑤ **전문이 snapshot에 텍스트로 담기나 + 중간 누락 없나**: 전문이 추출 가능한 텍스트로 나오나(이미지/캔버스 아님). **가상 스크롤로 중간 구간이 빠지지 않는지** — PageDown으로 끝까지 내려 마지막 섹션 종료 표지를 확인해야 '완전'으로 본다.
- [ ] ⑥ **총 건수 표시**: 검색결과 총 N건이 화면에 표시되나(완료 판정 기준값). **미표시면 누락 0을 보장 못 함 → 변호사에게 명시 보고**.
- [ ] ⑦ **비로그인 노출 범위**: (참고) 로그인 전엔 미리보기만 보이고 전문은 로그인 후인지.

### C-5. 검증 통과 후 다음 단계

1. 미확인 7개 확정값을 반영해 `case-researcher` 에이전트 정의를 마무리(페이지네이션 루프·라벨 추출·심급 묶음 로직 하드코딩).
2. (동의 시) `.mcp.json`에 `--user-data-dir ./.browser-profile` 적용 → 재로그인 불필요·기기 동일성 확보.
3. **본격 수집**: 같은 키워드를 §A-3 배치 상한(15~25건)으로 페이지네이션 끝까지 전수. worklist 재개 로직으로 여러 호출에 걸쳐 완주.
4. **Phase3/4 셋업**(별도): 저장된 .md에 원문 발췌+볼드(Phase3), docx 표 생성 도구(python-docx/pandoc 등 — 현재 미설치) 검증 후 도입(Phase4). 이 단계는 브라우저 미사용이라 병렬·오프라인 가능.

## D. 검증을 작게 유지하는 안전장치

- **우선 N건만**: 검증 기본 `검증건수=3`(전문 저장). 광범위 키워드면 1~3건으로 더 줄여 시작. 성공 확인 후에야 본격 전수로 확장.
- **트리아지도 첫 페이지 한정**: 검증 단계 Phase1은 전체 페이지가 아니라 **첫 1~2페이지**만. 전수 트리아지는 본격 단계에서.
- **하드 상한**: 한 검증 호출에서 **전문 열람 ≤ 5건**, **페이지 넘김 ≤ 2회**. 도달하면 무조건 저장·종료.
- **이상 시 즉시 중단 우선**: 차단 징후(§A-4)·비효율(§A-5)은 검증 단계에서 **더 민감하게** 적용(검증의 목적 자체가 "안전한지" 확인이므로). 의심되면 진행보다 보고를 택한다.
- **재개 가능성**: 모든 진척을 worklist에 즉시 기록 → 끊겨도 다음 호출이 worklist Read 후 미완료부터 이어감. 검증이 중간에 멈춰도 손실 없음.
- **사용자 승인 게이트**: (1) 영속 프로필 적용 전, (2) 검증→본격 전수 확장 전, (3) Phase4 docx 도구 설치 전 — 각각 사용자 동의를 받고 진행.

## 부록 — 저장 포맷·worklist 양식의 SSOT 위치

저장 파일(`data/cases/{사건번호}.md`)과 worklist(`data/cases/_worklist-{슬러그}.md`)의 **정본(SSOT)은 `case-researcher` 에이전트 정의**(`.claude/agents/case-researcher.md`)다. 이 문서에 중복 표를 두면 동기화가 깨지므로 여기서는 핵심 원칙만 남기고 양식 전문은 에이전트 정의를 본다.

- **저장 포맷(§2-3)**: 파일 = 판결 1건. `frontmatter`(Phase4 docx 컬럼에 1:1 매핑) + `## 0. 관련 슬라이스 포인터`(우리 판단·Phase3 작업지시) + `## 1. 전문`(**원문 무손실 보존, 잘라내지 않음**). (a)당사자/(b)범죄사실/(c)법리/(d)적용판단은 §1 안의 **위치를 §0이 지목**하며, 발췌·볼드는 Phase3에서만 한다.
- **worklist 양식**: 헤더(키워드/슬러그/대상 사실관계/검색 필터/총건수/페이지네이션 방식·뷰포트/현재 단계) + 진행률 요약 + Phase1 페이지 체크리스트 + 판례 인벤토리 표(16열: #/사건번호/사건명/법원/심급/선고일/죄목태그/유무죄(추정)/목록필터/전문검토/P2사유/확정여부/제외사유/심급그룹키/저장파일/lbox URL) + 중단·이상징후 로그. **제외 행도 삭제 금지**(완전성 검증·사후 재검토용).
- 목록필터 ∈ {통과, 보류(태그무), 제외(태그무)} / 전문검토 ∈ {미검토, 관련, 제외, 보류}. 모두 false-positive 우호(조금이라도 유사하면 통과·보류).
- **lbox_url·도메인**: 실제 존재하는 href만 기록(추측·조합 금지). 도메인 표기는 본 프로젝트에서 `〔실검증서 확정〕` 상태이며, 실검증 0단계에서 라이브로 관찰한 정확한 도메인을 CLAUDE.md·case-collector·새 에이전트·worklist 헤더·lbox_url에 통일한다.
